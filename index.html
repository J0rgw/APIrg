Obtener datos de partidas con la Riot API

Flujo correcto con match-v5 sobre la API pública: de un Riot ID a la timeline minuto a minuto, sin los errores típicos de routing y de eventos.

PASO 1
Riot ID → PUUID
account-v1
→
PASO 2
PUUID → match IDs
match-v5 /by-puuid
→
PASO 3
matchId → datos
match-v5 /{matchId}
→
PASO 4
matchId → timeline
match-v5 /timeline

PASO 0 · PREREQUISITOSAPI key y límites

Toda petición lleva la cabecera X-Riot-Token (preferible) o el query param ?api_key=. La development key se regenera cada 24 h; para producción se solicita una personal/production key.

Límites de la development key: 20 req/s y 100 req/2 min. Conviene meter un limitador (token bucket) desde el día uno, porque el coste real de poblar partidas son las dos llamadas por matchId (detalle + timeline).

PASO 1 · IDENTIDADDel Riot ID al PUUID

El PUUID es la clave que ata todo el flujo. Se obtiene del Riot ID moderno (gameName#tagLine) mediante account-v1. Es global y no cambia aunque el jugador cambie de servidor.

GET /riot/account/v1/accounts/by-riot-id/{gameName}/{tagLine}
Routing regional, no de servidor

account-v1 usa routing regional (americas / asia / europe), igual que match-v5, no los platform IDs como euw1. El dato devuelto es idéntico en cualquier cluster; solo cambia la latencia. Usa el más cercano a tu servidor.

PASO 2 · HISTORIALDel PUUID a los match IDs

Devuelve un array de strings tipo EUW1_7412345678. Es paginado: pides ventanas con start + count, o filtras por rango temporal y cola.

GET /lol/match/v5/matches/by-puuid/{puuid}/ids

Query params útiles

  • startint · def 0
    Offset de paginación.
  • countint · def 20 · máx 100
    Tamaño de página. Para ingesta masiva, itera de 100 en 100 subiendo start.
  • queueint
    Filtra por queue ID (p. ej. 420 = SoloQ ranked, 400 = normal draft). Útil para excluir ARAM/customs.
  • typestring
    ranked, normal, tourney, tutorial.
  • startTime / endTimeepoch segundos
    Acota por fecha. Ojo: segundos, no milisegundos.

PASO 3 + 4 · DATOSDetalle de partida y timeline

Por cada matchId hay dos recursos distintos. El detalle trae el estado final agregado (stats por participante, picks, bans, objetivos). La timeline trae la evolución temporal: snapshots por minuto y eventos puntuales.

GET /lol/match/v5/matches/{matchId}

Estructura: metadata (lista de PUUIDs) + info (los participants[] con KDA, oro, daño, items, runas; y teams[] con bans y objetivos). El orden de participants es el mismo en todo el JSON.

GET /lol/match/v5/matches/{matchId}/timeline

Estructura: info.frames[], un frame por minuto. Cada frame tiene participantFrames (posición, oro, XP, CS total y CS de jungla, nivel) y events[] (sucesos con timestamp en ms).

CLAVE #1Routing: región ≠ servidor

El error nº 1. El sufijo de EUW1_… en un matchId es un platform ID, pero match-v5 y account-v1 se consultan contra el cluster regional que lo engloba. Si pegas a euw1.api… para match-v5, falla.

Cluster regionalPlatform IDs (servidores)
americasna1 · br1 · la1 · la2
europeeuw1 · eun1 · tr1 · ru · me1
asiakr · jp1
seaoc1 · ph2 · sg2 · th2 · tw2 · vn2

Regla práctica: para todo lo de match-v5 y account-v1 usa el cluster (americas/europe/asia/sea). Los platform IDs solo aparecen en endpoints legacy como summoner-v4 o league-v4.

CLAVE #2Qué hay (y qué no) en la timeline

El error nº 2, y el más caro porque pasa silenciosamente: asumir eventos que la API no emite. La timeline tiene un conjunto cerrado de tipos en events[].type. Estos sí existen:

CHAMPION_KILL
CHAMPION_SPECIAL_KILL
BUILDING_KILL
TURRET_PLATE_DESTROYED
ELITE_MONSTER_KILL
WARD_PLACED / WARD_KILL
ITEM_PURCHASED / _SOLD / _DESTROYED / _UNDO
SKILL_LEVEL_UP
LEVEL_UP
GAME_END · PAUSE_END
Trampa de la jungla

No existe MONSTER_KILL para campamentos normales. Gromp, krugs, raptors, lobos y los buffs azul/rojo no generan ningún evento en la timeline. Riot llegó a emitir los buffs en su día y los retiró a propósito (evitar exponer rutas y counter-jungla). Lo único que llega como evento de monstruo es ELITE_MONSTER_KILL: dragones, Barón, Heraldo, Grubs/Voidgrubs y Atakhan.

Filtrar la ruta de jungla por un tipo de evento que no se emite "funciona" contra fixtures inventados y devuelve vacío contra datos reales. La ruta de campamentos no se lee de events: se infiere de los snapshots.

Para objetivos épicos sí usa los eventos: ELITE_MONSTER_KILL trae monsterType, monsterSubType (tipo de dragón), killerId y position, que es justo lo que quieres para timings y control de objetivos.

CLAVE #3IDs numéricos → nombres

Casi todo lo que identifica contenido viene como entero: championId en picks, los IDs de campeón en bans, queueId, items, etc. Para mostrarlos legibles hay que resolverlos contra Data Dragon (versionado por parche) o Community Dragon.

Guarda el mapa por parche: los IDs son estables, pero resolver nombres en cada request es trabajo tirado.
